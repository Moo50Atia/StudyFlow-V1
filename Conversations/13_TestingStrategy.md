# 13. Quality Assurance & Testing Strategy Specification

---

## 1. Testing Architecture & Principles

### 1.1 Testing Pyramid
The platform's verification structure MUST balance execution speed, coverage, and costs across five layers:

```
                  / \
                 /   \
                / AI  \  <-- Semantic validation, coverage checks (Stage 10)
               /-------\
              /   E2E   \ <-- Full pipeline runs (from upload to pub)
             /-----------\
            /  Contract   \ <-- DTOs, JSON Contracts, API payload schemas
           /-------------\
          /  Integration  \ <-- Subprocess calls, LTI launches, db transactions
         /-----------------\
        /      Unit        \ <-- Class methods, chunking formulas, helper files
       /---------------------\
```

### 1.2 Testing Principles
* **Offline Isolation (Cost Efficiency):** Normal unit and integration test suites MUST run without internet access, utilizing local mocks for external LLM API calls to avoid token billing charges.
* **Repeatability:** Test assertions MUST be deterministic, eliminating random data or floating thresholds.
* **Coverage Gates:** Code changes MUST maintain minimum verification criteria before merge authorization.

---

## 2. Test Environment Strategy

* **2.1 Local Development Environment:** Utilizes local mock data services. Developers run localized SQLite or in-memory databases to ensure rapid feedback cycles.
* **2.2 Continuous Integration (CI) Gates:** Automated test suites executed on pull requests. The CI environment MUST run static analysis, contract compliance checks, and security vulnerability scanners, blocking merges on any failure.
* **2.3 Staging Environment:** Real LLM API integration tests, performance profiling, and sandbox LTI connections to Canvas/Moodle. Staging is used to run the active AI Evaluation suite.
* **2.4 Production Verification:** Live smoke tests, execution log health checks, and correlation ID tracing audits on canary deployments.

---

## 3. Test Data Management

* **3.1 Static Fixtures & Mock JSON:** Immutable JSON files simulating outputs from every pipeline stage (e.g. mock `structure.json`, `sections.json`) to test downstream sync services.
* **3.2 Golden Datasets & Sample PDFs:** A curated repository of representative university textbooks and lecture notes with known, verified target structures, questions, and Arabic analogies.
* **3.3 Snapshots:** Visual and JSON snapshots to track output drift during code modifications.

---

## 4. Contract Testing Strategy

* **4.1 JSON Contract Compatibility:** Assertions MUST verify that JSON outputs generated by the Python pipeline validate against the contracts defined in [05_JSONContracts.md](file:///D:/projects/laravel_projects/college_project/Conversations/05_JSONContracts.md).
* **4.2 REST API Integration Verification:** Asserts that HTTP request payloads and API resource responses comply with versioned contracts in [06_APISpecification.md](file:///D:/projects/laravel_projects/college_project/Conversations/06_APISpecification.md).
* **4.3 Laravel DTO Verification:** Unit tests MUST assert that parsing JSON strings into DTO objects results in correct data types and does not drop fields.

---

## 5. Subsystem Verification Plans

### 5.1 Laravel Unit & Feature Testing
* **Persistence Transactions:** Test suites MUST execute within database transaction boundaries, reverting all changes after each test.
* **Route Permissions:** Verify that the API returns `403 Forbidden` if a teacher attempts to edit resources in a course they do not own in `teacher_permissions`.

### 5.2 Python Unit & Integration Testing
* **Stage Script Isolations:** Test each Python module (e.g. `chunking`, `ocr`) with dedicated input arguments, validating outputs without executing the entire pipeline.
* **Process Harness Checks:** Assert that `pipeline.py` correctly handles CLI flags (`--start-from`, `--stop-after`) and returns appropriate status codes.

---

## 6. AI Evaluation & Quality Assurance

### 6.1 AI Evaluation Dataset Strategy
The system MUST maintain a stable, versioned set of benchmark academic texts (the "AI Benchmarks") to track output quality. If model prompts or LLM engines are updated, the benchmarks MUST be re-processed, and output sections compared for semantic drift.

### 6.2 Semantic Coverage Verification
Integration tests MUST assert that Stage 10 (`validate`) reports a curriculum coverage score above 85% and a visualization readiness score above 60% when run on benchmark datasets.

### 6.3 Hallucination Checking
Automated assertion checks MUST compare generated sections against source text using keyword checks, validating that no new mathematical formulas or clinical facts are introduced.

---

## 7. Dynamic View Verification & Compatibility

### 7.1 Headless Sandbox Render Checks
* Stage 12 verification MUST trigger a headless browser runtime instance to load the compiled HTML asset.
* The test MUST assert:
  * No unhandled JavaScript exceptions are thrown on load.
  * The DOM contains all widget IDs mapped in `dynamic_view_mapping.json`.
  * The canvas is responsive and adjusts scaling boundaries.

### 7.2 LMS / LTI Interoperability Testing
The integration test suite MUST mock LTI OIDC redirects, JWT claims, and keyset endpoints to verify that the Web portal handles secure launches from Canvas, Moodle, and Blackboard configurations.

---

## 8. Non-Functional Testing Strategy

* **8.1 Performance & Load Testing:** Load testing MUST verify that the API Gateway can process 500 concurrent LTI launches and telemetry log requests while keeping page render times under 1.5 seconds.
* **8.2 Stress & Scalability Testing:** Verify pipeline execution when ingesting large PDFs (e.g. 500+ pages) or running concurrent ingestions, asserting that system memory usage does not exceed limits.
* **8.3 Memory Leakage Validation:** Memory profiling tools MUST inspect long-running ingestion processes to detect memory leaks in Tesseract OCR bindings or Python subprocess execution loops.

---

## 9. Security Testing Strategy

* **9.1 RBAC Authorization Verification:** Asserts that student sessions are blocked from executing modification endpoints (e.g. PUT, POST, DELETE).
* **9.2 Tenant Data Isolation:** Assertions MUST verify that queries fail if a request attempts to fetch resource IDs mapping to a different tenant context.
* **9.3 JWT & Token Validation:** Verify that expired, forged, or unsigned JWT payloads are rejected with `401 Unauthorized`.
* **9.4 File Upload Security:** Test that uploading non-PDF files, executable binaries, or oversized payloads results in a validation failure and rejects files.
* **9.5 Path Traversal Prevention:** Verify that relative paths (e.g. `../../etc/passwd`) passed in subject name inputs are escaped, preventing path traversal attacks.

---

## 10. Regression & Mutation Testing

* **10.1 Material Regression Prevention:** System integration tests MUST verify that changes to the Python pipeline do not break rendering compatibility for previously compiled HTML views.
* **10.2 Mutation Testing:** The deployment pipeline SHOULD optionally execute mutation testing (systematically altering logic operator signs in code) to verify that the test suite catches altered logical paths.

---

## 11. Acceptance & Observability Validation

* **11.1 Acceptance Criteria:**
  * **Professor Acceptance:** The HIL Node Editor compiles correctly, saves edits, and resumes the pipeline.
  * **LTI Acceptance:** Scores write back to the LMS Gradebook with success logs.
  * **Business Acceptance:** Total processing costs and time metrics map parameters defined in business strategies.
* **11.2 Observability Validation:** Integration tests MUST assert that a simulated run generates logs matching the active `correlation_id` across the API, Queue, and Python output logs.
