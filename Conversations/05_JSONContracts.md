# 05. JSON Contracts Specification

This specification defines the formal, implementation-agnostic data structures exchanged between the ingestion pipeline stages and the persistence services. 

---

## 1. Document Control & Versioning Policy

### 1.1 Schema and Contract Versioning
All schemas defined in this specification MUST be versioned using Semantic Versioning (Major.Minor.Patch). 
* **Major Version Changes:** Introduce breaking alterations where attributes are removed, renamed, or types are modified. Major changes require consumer-side parser updates.
* **Minor Version Changes:** Introduce non-breaking alterations, such as optional fields or extensible choices.
* **Patch Version Changes:** Clarify descriptions or documentation without structural changes to attributes.

### 1.2 Backward and Forward Compatibility Rules
* **Backward Compatibility:** Future versions of a contract MUST NOT remove existing fields or mark previously required fields as optional. Consumers written for older versions MUST be able to parse newer contracts by ignoring unrecognized attributes.
* **Forward Compatibility:** Parsers built to consume newer versions of a contract MUST define fallback behaviors or ignore missing non-required fields when reading older versions of the same file.

### 1.3 Artifact Identity, Metadata, and Integrity
Every JSON artifact generated by the pipeline MUST include a standardized root metadata envelope:

* **Artifact Identity:**
  * `artifact_id`: A unique string identifying the specific file instance.
  * `run_id`: The unique execution pipeline identifier mapping this artifact to its run context.
* **Artifact Metadata:**
  * `created_at`: UTC ISO-8601 timestamp.
  * `schema_version`: Semantic version string.
  * `artifact_name`: Filename matching those in the pipeline stages.
* **Artifact Checksums:**
  * `checksum`: A SHA-256 hash of the JSON content (excluding the checksum field itself) computed during compile-time.
* **Artifact Provenance & Integrity Verification:**
  * `provenance`: A structural history log documenting modifications, including the owner ID, phase stage name, and parent artifact hashes.
  * *Verification Rule:* Consumers MUST verify that the calculated SHA-256 matches the `checksum` field before synchronization or validation execution.

## 2. Common Data Types & Polymorphic Contracts

### 2.1 Canonical Mathematical Representation
All equations and mathematical expressions within the system MUST use a structured canonical object definition:
* **`formula_type`:** Enum indicating representation style (e.g. `LaTeX`, `MathML`, `AsciiMath`).
* **`expression`:** The raw formula text.
* **`alternatives`:** Optional key-value mappings containing alternative representations for different renderers.

### 2.2 Polymorphic Route-Specific Sub-Schemas
Artifacts containing route-specific pedagogical fields (e.g., `sections.json`, `dynamic_view_mapping.json`) MUST deploy a polymorphic schema design:
* **The Base Entity:** Contains global structural and routing variables.
* **Polymorphic Attribute Envelope:** Maps attributes dynamically based on the detected `route` string.
* **Validations:** A client parsing a `Medical` route content node MUST expect `clinical_scenario` and perspective arrays, whereas a client parsing a `Mathematics` route content node MUST expect equation formula blocks.

## 3. Extraction & Chunking Phase Contracts

### 3.1 `extraction_metadata.json`
* **Purpose:** Stores the physical layout measurements and raw page indexes parsed from the PDF.
* **Fields:**
  * `total_pages`: Total integer page count.
  * `page_dimensions`: Coordinates (width, height) of pages.
  * `layout_blocks`: Array containing coordinates, page bounds, and structural offsets.

### 3.2 `ocr_report.json`
* **Purpose:** Stores structural and validation reports for OCR conversion.
* **Fields:**
  * `processed_pages`: Array of integers listing pages processed via image extraction.
  * `confidence_scores`: List of floats representing OCR recognition accuracy per page.
  * `density_measurements`: Page character densities.

### 3.3 `chunk_manifest.json`
* **Purpose:** Maps text divisions and boundaries.
* **Fields:**
  * `chunks`: Array of chunk objects, each containing a chunk identifier, string boundary indices (`start_index`, `end_index`), and page mapping arrays.

## 4. Routing & Structure Mapping Contracts

### 4.1 `route_detection.json`
* **Purpose:** Records course domain classifications.
* **Fields:**
  * `route`: Classified route string.
  * `confidence`: Classification confidence value.
  * `keyword_frequencies`: Array of keyword hit metrics.

### 4.2 `material_config.json`
* **Purpose:** Stores pipeline configuration parameters and overrides.
* **Fields:**
  * `material_name`: The unique identifier folder name of the course run.
  * `mode`: Processing mode string.
  * `require_approval`: Boolean override gating Stage 6.

### 4.3 `structure.json`
* **Purpose:** Hierarchical representation of the curriculum outline.
* **Fields:**
  * `chapters`: Array of Chapter nodes.
  * `chapters[].id`, `chapters[].title`: Chapter details.
  * `chapters[].mini_chapters`: Array of mini-chapter nodes.
  * `chapters[].mini_chapters[].lessons`: Array of lesson nodes.
  * `lesson_node`: Contains identifiers, titles, page boundaries (`page_start`, `page_end`), and source references.

## 5. Pedagogical Rewrite & Validation Contracts

### 5.1 `knowledge_graph.json`
* **Purpose:** Maps conceptual dependencies and definitions.
* **Fields:**
  * `entities`: Array of nodes (definitions, equations, topics).
  * `edges`: Array of dependency linkages connecting entity nodes.

### 5.2 `questions.json`
* **Purpose:** Stores the extensible bank of practice checks.
* **Fields:**
  * `questions`: Array of extensible question objects.
  * `base_question`: Contains ID, question text, type flag (`question_type`), and mapped topic.
  * `polymorphic_details`: Maps requirements dynamically:
    * `multiple_choice`: Array of option strings, correct option index.
    * `fill_blank`: Array of regex pattern strings, case sensitivity flags.
    * `open_ended`: Suggested scoring keyword checklist arrays.

### 5.3 `sections.json`
* **Purpose:** Holds pedagogically rewritten lesson sections.
* **Fields:**
  * `sections`: Base section models containing title, core concept, summaries, and localized analogies (`arabic_explanation`).
  * `polymorphic_enrichment`:
    * `Medical`: Contains clinical scenarios, clinical check questions, and medical term translations.
    * `Mathematics`/`Physics`: Contains formulas matching the Canonical Mathematical Representation.

### 5.4 `validation.json`
* **Purpose:** Records curriculum coverage analysis.
* **Fields:**
  * `coverage_percentage`: Calculated content mapping score.
  * `visualization_readiness`: Calculated visualization capability score.
  * `omitted_topics`: Array of topics or pages flagged as unmapped.

## 6. Dynamic View Rendering & Manifest Contracts

### 6.1 `dynamic_view_mapping.json`
* **Purpose:** Maps sections to structured interactive scenes.
* **Fields:**
  * `mappings`: Array of section-to-canvas definitions.
  * `mapping_node`: Contains section link ID, unique target view name, and dynamic view category (e.g. Process, Accumulation).
  * `visual_objects`: Array of strongly typed canvas widgets:
    * `object_id`: Unique identifier string.
    * `object_type`: Widget type (e.g. slider, dynamic chart, switch).
    * `properties`: Key-value properties representing values, units, step ratios, boundaries, and variables.
  * `interactions`: Action-trigger maps linking interface inputs to visual state changes.

### 6.2 `dynamic_view_prompts.json`
* **Purpose:** Holds compiled prompting scripts.
* **Fields:**
  * `prompts`: Array of prompt nodes linking unique section IDs to visual rendering scripts and target file destination values.

### 6.3 `dynamic_view_cache.json`
* **Purpose:** Caches pre-generated UI templates and layouts.
* **Fields:**
  * `cached_views`: Hash array of section IDs to compiled HTML templates.

### 6.4 `manifest.json`
* **Purpose:** Ingestion metadata summaries.
* **Fields:**
  * `run_id`: Unique string mapping the execution lifecycle.
  * `model_identifier`: Identifiers indicating which model executed the run.
  * `prompt_tokens`: Count of input prompt tokens.
  * `completion_tokens`: Count of output tokens generated.
  * `timestamps`: Execution timestamp logs (`started_at`, `completed_at`).
  * `execution_duration`: Active processing duration in seconds.
  * `artifacts`: Key-value pairs matching artifact filenames to verification hashes and output locations.

## 7. Pipeline Stage Error Contracts

### 7.1 Standard Stage Failure Schema
If a pipeline stage encounters a terminal exception, the error artifact written to disk MUST follow this structure:
* **`error_code`:** Unique classification string (e.g. `ERR_JSON_PARSE`, `ERR_LLM_TIMEOUT`).
* **`failed_stage`:** Name of the stage that threw the error.
* **`error_message`:** Description of the error.
* **`stack_trace`:** Debugging stack trace logs.
* **`unvalidated_payload`:** Optional raw text block showing the unvalidated output payload before the crash occurred.
